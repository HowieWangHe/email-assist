# 企业邮箱调研/询价助手项目设计

日期：2026-05-03

## 1. 项目目标

构建一个本地 Web 项目，用于托管企业邮箱的调研/询价工作流。用户提供企业邮箱的 SMTP/IMAP 地址、端口、账号和协议配置后，系统支持批量向多个收件人发送调研/询价邮件，追踪收件人回复，归档邮件正文和附件，并在收齐回复后对附件数据进行结构化处理。

第一版定位为单人本地使用的 MVP，优先保证端到端流程可用、状态可追踪、文件可归档、结果可导出。

## 2. 已确认范围

### 2.1 第一版包含

- 本地 Web 单体应用。
- 使用 SMTP 发信，使用 IMAP 收信和追踪回复。
- 支持联系人库、CSV/Excel 导入、手动粘贴或输入收件人。
- 外部收件人逐个单独发送，彼此不可见。
- 支持全局 CC/BCC，用于内部同事、项目组邮箱或归档邮箱。
- 支持手动刷新收件箱和项目运行期间定时轮询。
- “回执”定义为收件人回复了邮件，不依赖已读回执或送达回执。
- 支持识别全部目标收件人是否已回复。
- 支持截止时间前 6 小时对未回复收件人催发提醒邮件。
- 催发策略可按任务配置：自动催发或人工确认催发，默认人工确认。
- 支持本地规则摘要邮件正文。
- 支持附件按任务物理归档，系统按供应商/联系人检索。
- 支持附件 AI API 处理，接入 OpenAI-compatible API。
- 附件处理支持 Excel/CSV、PDF/Word、图片/OCR。
- 附件抽取支持内置询价/报价模板和自定义字段模板。
- 支持导出 Excel 汇总和 ZIP 任务包。

### 2.2 第一版不包含

- 多用户权限系统。
- 云端部署。
- 强依赖已读回执或送达回执。
- 默认自动调用 AI 处理所有邮件正文。
- 邮件内容、附件、联系人数据的加密存储。
- 复杂消息队列系统。
- 面向 500 个以上收件人的大规模群发优化。

## 3. 可行性结论

该项目可行，推荐使用 `FastAPI + SQLite + 本地文件存储 + 本地 Web UI` 实现。

SMTP/IMAP 可以稳定覆盖发信、收信、轮询、附件下载和回复追踪。传统邮件的已读回执和送达回执不可靠，因此第一版不把它们作为状态依据。系统通过每个调研任务的任务编号、主题前缀、发件记录、`Message-ID`、`In-Reply-To`、`References`、发件人邮箱等信息识别收件人是否回复。

附件 AI 处理可行，但附件内容调用外部 AI API 时会离开本机。第一版必须通过任务级开关明确启用，并记录调用配置、模型、处理结果和失败信息。

## 4. 推荐方案

采用方案 A：本地 Web 单体 MVP。

技术建议：

- 后端：FastAPI
- 数据库：SQLite
- 文件存储：本地目录
- 邮件发送：SMTP
- 邮件接收：IMAP
- 前端：轻量 Web UI，优先考虑 Jinja/HTMX 或类似低复杂度方案
- AI 接入：OpenAI-compatible API，配置 `base_url`、`api_key`、`model`

该方案开发快、部署简单、适合本地单人使用。内部保持模块化边界，后续可升级前端、增加后台队列或扩展多人协作。

## 5. 总体架构

核心模块：

- 账号配置模块：保存 SMTP、IMAP、AI API 配置，支持连接测试。
- 联系人模块：维护联系人库，支持 CSV/Excel 导入和手动输入。
- 调研任务模块：管理任务名称、主题、正文模板、截止时间、催发策略、附件处理模板。
- 发信模块：对外部收件人逐个单发，记录每封邮件的追踪信息。
- 收信追踪模块：手动刷新或定时轮询 IMAP，匹配回复邮件。
- 催发模块：在截止时间前 6 小时检查未回复对象，并按任务策略催发。
- 附件归档模块：保存回复正文、原始邮件和附件。
- 附件 AI 处理模块：处理 Excel/CSV、PDF/Word、图片/OCR，并按模板抽取字段。
- 导出模块：生成 Excel 汇总和 ZIP 任务包。

关键边界：

- 本地服务停止期间不会实时追踪；重启后可补拉邮箱并补检查仍在合理窗口内的催发任务。
- 已读回执和送达回执不作为可靠状态依据。
- 附件内容调用 AI API 时必须由任务级开关启用。

## 6. 核心业务流程

### 6.1 创建任务

用户创建调研/询价任务，填写：

- 任务名称
- 邮件主题
- 正文模板
- 截止时间
- 外部收件人
- 全局 CC/BCC
- 邮件附件
- 催发策略
- 是否开启附件 AI 处理
- 附件抽取模板

收件人来源：

- 联系人库选择
- CSV/Excel 导入
- 手动粘贴或输入

### 6.2 发送邮件

系统为每个外部收件人单独发送一封邮件，内部人员可作为全局 CC/BCC。

每封邮件记录：

- 任务编号
- 收件人
- 主题前缀
- Message-ID
- 发送时间
- 发送结果
- 附件清单

### 6.3 追踪回复

系统通过手动刷新或定时轮询 IMAP 拉取新邮件。回复匹配优先级：

1. 使用 `In-Reply-To` / `References` 匹配已发送邮件的 `Message-ID`。
2. 使用发件人邮箱 + 任务编号或主题前缀匹配。
3. 低置信度匹配进入待人工确认。

### 6.4 自动催发

到 `截止时间 - 6 小时` 时，系统检查仍未回复的外部收件人。

- 自动催发：直接发送提醒邮件。
- 人工确认催发：生成提醒草稿或待办，用户确认后发送。
- 默认每个任务每个收件人最多催发一次。
- 如果本地服务停机错过窗口，重启后只对仍未截止且未催发的任务补处理。

### 6.5 收齐回复

当所有目标外部收件人都已回复或被人工确认回复后，任务进入 `all_replied`。发送失败、人工排除或标记为无需回复的联系人不计入待收齐分母。

### 6.6 附件归档和正文摘要

收到回复后，系统保存：

- 原始邮件 `.eml`
- 邮件头
- 正文 `.txt`
- 本地规则摘要
- 附件
- 附件元数据

附件按任务和收件人归档，数据库提供按供应商/联系人检索的索引。

### 6.7 附件 AI 处理

如果任务开启附件 AI 处理，且任务进入 `all_replied`，系统触发处理流程：

- Excel/CSV：直接读取表格并结构化抽取。
- PDF/Word：先抽取文本，再交给 AI 结构化。
- 图片：先 OCR 或视觉模型识别，再结构化。
- 无法解析或处理失败的文件进入异常清单。

输出包括：

- 内置询价/报价模板字段
- 自定义字段模板结果
- 每个附件的原始抽取结果
- 异常和缺失项
- 汇总 Excel
- JSON 明细
- Markdown 报告

## 7. 状态设计

### 7.1 收件人状态

- `draft`：待发送
- `sent`：已发送未回复
- `replied`：已确认回复
- `needs_review`：疑似回复，待人工确认
- `failed`：发送失败
- `overdue`：超过截止时间仍未回复
- `reminded`：已催发但仍未回复

### 7.2 任务状态

- `draft`：草稿
- `sending`：发送中
- `tracking`：追踪中
- `reminder_pending`：有催发待确认
- `all_replied`：全部已回复
- `processing_attachments`：附件处理中
- `completed`：已完成
- `closed`：人工关闭
- `error`：存在阻塞错误

## 8. 数据模型

SQLite 主要数据表：

- `mail_accounts`：SMTP/IMAP 配置、发件邮箱、默认 CC/BCC、轮询间隔。
- `contacts`：姓名、邮箱、公司、电话、标签、备注。
- `campaigns`：任务名称、主题、正文模板、截止时间、催发策略、附件处理开关、状态。
- `campaign_recipients`：任务收件人、发送状态、回复状态、催发状态、错误原因、发送时间、回复时间。
- `sent_messages`：每封已发送邮件的 `Message-ID`、收件人、主题、邮件头、附件清单。
- `received_messages`：匹配到的回复邮件、正文路径、邮件头、匹配置信度、确认状态。
- `attachments`：附件原始文件名、保存路径、来源邮件、文件类型、解析状态、失败原因。
- `field_templates`：内置报价模板和自定义字段模板。
- `extraction_results`：每个附件或供应商的字段抽取结果、置信度、异常项。
- `reminder_logs`：催发记录、催发时间、方式、结果。
- `job_logs`：收信、发信、附件处理、导出的任务日志。

## 13. 异常处理

- SMTP 发送失败：记录错误原因，允许单个收件人重试。
- IMAP 连接失败：保留上次检查时间，不覆盖已有状态。
- 回复匹配低置信度：进入待确认，不自动计入已回复。
- 附件重名：保留原始文件名，同时用内部唯一 ID 防止覆盖。
- 附件解析失败：保留原文件，写入异常清单。
- AI API 失败：记录请求目标、模型、错误信息、重试次数，不丢失本地解析结果。
- 自动催发：每个任务每个收件人默认最多一次，并记录日志。
- 服务停机：重启后补拉邮件；催发只在未截止且未催发的合理窗口内补做。

## 14. 隐私与安全边界

- 第一版本地明文保存邮件正文、附件、联系人和任务数据。
- 邮箱密码和 API Key 不写入代码文件，不提交仓库。
- 本地配置文件需要加入 `.gitignore`。
- AI API 处理附件时，附件内容或抽取文本会发送到配置的外部 API。
- 后续可扩展系统钥匙串、本地加密、字段级脱敏和调用审计。
